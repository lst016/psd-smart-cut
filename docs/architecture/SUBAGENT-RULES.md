# Subagent 强制行为准则

本文定义 `PSD Smart Cut V2` 中所有 `subagent` 必须遵守的操作规则。

这些不是建议，而是强制约束。

## 核心定位

`subagent` 首先是页面理解者、规划者和审阅者，其次才是执行者。

它们不允许退化成“看到图层就导出”的盲切工具。

所有子 agent 都必须基于以下顺序思考：

1. 先看截图，理解页面用途
2. 再判断业务模块和子页面关系
3. 再按前端组件逻辑拆解
4. 再按美术复用逻辑规划资源
5. 最后才允许进入导出

PSD 结构只能作为辅助证据。

## 不可协商的规则

### 1. 截图优先

在决定组件边界前，必须先理解整页截图。

禁止直接从以下内容开始：

- 原始图层名
- group 名
- 面积阈值
- 规则匹配结果

如果没有先形成页面级理解，不允许进入组件拆解。

### 2. 先模块，后组件

任何 `subagent` 在拆组件之前，必须先判断：

- 这页是干什么的
- 有几个业务模块
- 哪些是子页面
- 哪些是主区域，哪些是次级区域

例如：

- 视觉上看到 3 个画面，不代表就是 3 个独立业务页面
- Android 和 iOS 可能只是同一个教学模块的两个子页面

### 3. 前端意义高于 PSD 图层意义

默认优先采用前端实现逻辑，而不是 PSD group 结构。

错误行为：

- 一个 group 就当一个 component
- 一个重复按钮实例就导出一张独立位图
- 一个 switch 行就切成一张截图

正确行为：

- 识别共享状态资源
- 识别共享视觉皮肤
- 避免把实例级重复放进资源层

### 4. 复用优先

默认优先共享资源，而不是实例级导出。

常见默认策略：

- 按钮组：共享 `normal` / `selected`
- switch：共享 `active` / `inactive` / `thumb`
- tab：共享状态底图，不导出每个 tab 实例位图

除非确认存在视觉差异，否则禁止把 6 个相同按钮导成 6 张图。

### 5. 先组件计划，后最终导出

在申请导出图片之前，每个 `subagent` 必须先说清楚：

- 组件是什么
- 组件有哪些子部分
- 哪些文字保持可编辑
- 哪些资源是共享的
- 组件有哪些状态

如果这些问题没回答清楚，不允许直接导图。

### 6. PSD 结构只是辅助证据

PSD 结构可以用于：

- 对齐 bounds
- 核对分组关系
- 找 source layer
- 辅助解释视觉区域

但它不能被当成产品模型的主真相。

### 7. 必须显式留下文档证据

每个 `subagent` 都必须留下结构化结果，至少包括：

- 页面理解
- 模块树
- 组件计划
- 前端分析
- 实现决策
- 资源计划

不能只吐一个图片目录就算完成。

## 角色级强约束

### Product Subagent

必须回答：

- 页面用途是什么
- 有哪些业务模块
- 哪些是主流程，哪些是辅助信息
- 哪些页面是子页面，哪些是独立页面

禁止：

- 在模块树还没建立前就直接定义资源

### Frontend Subagent

必须回答：

- 有哪些可复用组件
- 这些组件有哪些状态
- 哪些文字应由前端渲染
- 哪些是组件实例，哪些是共享资源
- 哪些关键子部件应走 `image / css / text / svg`

禁止：

- 默认把重复实例导出成独立资源
- 没有必要时把文字当图片
- 用 PSD 图层边界直接替代组件边界

特别规则：

1. `switch` 默认拆为：
   - `track_active`
   - `track_inactive`
   - `thumb`
   - 行级实例配置
2. 底部按钮组默认拆为：
   - `button_normal`
   - `button_selected`
   - 多个按钮实例配置
3. 普通按钮、tab、说明文案优先保留为文本槽位

### Art Subagent

必须回答：

- 哪些内容必须保留为视觉资源
- 哪些资源可以跨实例复用
- 哪些背景可合并成共享皮肤

禁止：

- 没有明确理由时把整页大块烘焙成巨型背景图
- 忽略可复用的皮肤和装饰资源

### Export Subagent

必须回答：

- 具体要渲染哪些资源
- 应使用哪些稳定 id 和命名
- 组件如何引用资源

禁止：

- 在导出后再临时发明资源身份
- 把组件身份和资源身份混成一个 id

### Review Subagent

必须回答：

- 哪些内容过度导出
- 哪些内容漏切
- 哪些资源重复
- 哪些输出不符合前端实现逻辑

禁止：

- 仅因为输出目录结构完整就判定通过

## 必答审查问题

每次交付前，都必须回答以下问题：

1. 是否先基于截图理解了页面
2. 是否先识别了模块，再拆组件
3. 是否优先使用了可复用资源
4. 是否尽可能保留了可编辑文本
5. 是否区分了页面、组件和资源三种身份

只要有一项答案是 `否`，本轮结果都应视为未完成。

## 前端分析强制文件

凡是进入切图规划的任务，默认应先产出：

- `analysis/frontend-analysis.md`
- `analysis/frontend-analysis.json`
- `analysis/implementation-decision.json`

如果没有这三份文件，就不应直接进入 `resource-task-list`。
