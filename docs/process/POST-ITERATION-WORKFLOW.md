# 试运行后项目流

本文定义每次真实 PSD 试运行结束之后，项目应该如何收敛、复盘、精修并决定是否进入下一轮。

这份文档解决的问题是：

- 一次试运行结束后，下一步先做什么
- 什么阶段继续探索，什么阶段开始收敛
- 什么情况下可以继续换 PSD，什么情况下必须先补能力
- 如何避免“每轮都重新试错”

## 总原则

每次试运行之后，不允许直接跳到下一份 PSD。

必须先经过：

1. 结果归档
2. 失败点复盘
3. 资源类型归类
4. 策略收敛
5. 白名单精修
6. 生产准入判断

只有当本轮目标已经达到预期，或者失败原因已经被明确归档并转成工程任务，才允许进入下一轮。

## 标准后续链路

```text
试运行包
-> 复盘
-> 问题分类
-> 策略收敛
-> 白名单精修
-> 回归验证
-> 生产准入判断
-> 决定继续当前 PSD / 切下一份 PSD
```

## 阶段 1：结果归档

目标：

- 把本轮试运行包固定下来
- 保留页面分析、模块树、组件计划、资源任务单和导出结果

必须有：

- `analysis/page-analysis.md`
- `analysis/module-tree.md`
- `analysis/component-plan.md`
- `analysis/resource-task-list.json`
- `analysis/review-summary.md`
- `resources/<format>/index.json`

规则：

- 归档的是试运行结果，不是生产结果
- 没有归档，不允许直接口头进入下一轮

## 阶段 2：复盘

目标：

- 明确本轮哪些判断是正确的
- 明确本轮哪些导出不可进入生产
- 明确失败根因属于哪一层

复盘时必须回答：

1. 页面理解是否正确
2. 模块边界是否正确
3. 组件规划是否正确
4. 资源规划是否正确
5. 切图策略错在什么地方
6. 当前结果为什么还不能交付

输出建议：

- `docs/ITERATION-RETRO-<name>.md`

## 阶段 3：问题分类

目标：

- 不把所有失败都当成“切图有点问题”
- 把问题分类到正确的工程层

建议分类：

- `页面理解问题`
- `模块规划问题`
- `组件建模问题`
- `资源类型分流问题`
- `导出策略问题`
- `白名单缺失问题`
- `生产准入失败`

规则：

- 每个问题至少要回答一次“以后如何避免重复发生”

## 阶段 4：策略收敛

目标：

- 把本轮暴露出来的问题，收敛成更稳定的默认策略

本阶段必须做的事：

1. 先判断哪些资源其实已经可以单层直出
1. 更新资源类型分类
2. 更新默认导出策略
3. 更新 fallback 边界
4. 明确哪些问题不再继续通用探索

示例：

- `export_ready_single_layer` 优先单层直出
- `single_layer_icon` 优先单层直出
- `explicit_layer_combo` 优先显式层组合
- `composed_icon` 优先结构组合
- `context_state_asset` 优先上下文渲染或差分
- `visual_matting` 只保留为最后 fallback

规则：

- 一旦某类方案已经被证明不稳定，就不应在下一轮继续作为主路线大面积尝试
- 对任何资源，不应在没有做“单层直出探测”的情况下直接升级到复杂策略

## 阶段 5：白名单精修

目标：

- 对无法靠通用策略稳定处理的资源，引入白名单精修

典型场景：

- 底部栏图标
- 复杂状态资源
- 大厅卡片类资源
- 广告和强视觉模块

白名单至少应定义：

- `resource_key`
- `resource_type`
- `preferred_strategy`
- `fallback_strategy`
- `review_focus`

对于“多图层嵌套、但背景层并不属于目标资源”的场景，建议优先使用：

- `preferred_strategy: hide_background_then_render`
- `background_layer_ids`
- 可选 `context_layer_ids`

这类策略的核心不是后处理抠图，而是先识别背景贡献层，再隐藏背景后重新渲染目标资源。

规则：

- 白名单不是补丁，而是进入生产前的正式治理手段
- 如果某类资源反复失败，必须进入白名单

## 阶段 6：回归验证

目标：

- 验证策略收敛和白名单精修是否真的改善了结果

至少要做：

1. 当前 PSD 回归
2. 历史样例回归
3. 关键资源抽样对比

必须检查：

- 是否带入背景
- 是否丢边缘
- 是否把互斥态叠进同一资源
- 是否把实例资源错误重复导出
- 是否符合前端和美术的复用方式

## 阶段 7：生产准入判断

目标：

- 判断当前能力是继续工程探索，还是可作为生产交付链的一部分

只有同时满足以下条件，才可视为生产可用：

1. 资源已完成类型分流
2. 关键资源已有稳定导出策略
3. 白名单机制已覆盖反复失败的资源
4. 导出结果达到前端和美术可直接使用
5. 至少一轮回归验证通过

只要有一项不满足，就应标记为：

- `试运行`
- `方法验证`
- `工程探索`

不能标记为：

- `生产可用`

## 何时换下一份 PSD

可以换下一份 PSD 的情况：

1. 当前样例已经达到预期
2. 当前样例虽未达到预期，但失败原因已经被清晰归档
3. 下一轮要验证的是新的模块类型，而不是重复同一问题

不应换下一份 PSD 的情况：

1. 当前失败原因还说不清楚
2. 同一类资源问题还在重复试错
3. 白名单和资源类型策略还没更新

## 这份文档在项目中的作用

这份文档把每次试运行后的工作流固定下来，避免后续再次出现：

- 刚出结果就直接换新 PSD
- 失败后只做局部修补，没有形成策略
- 每轮都重复探索同一类问题
- 无法判断项目到底处在“探索”还是“可交付”阶段
## 新增策略补充

对真实 PSD 的后续治理，新增一条硬规则：

1. 对候选资源先做 `export_ready_single_layer` 探测。
2. 如果该层可直接导出，再检查是否存在启用中的图层效果。
3. 若存在 `ColorOverlay / GradientOverlay` 等效果，导出器必须先做效果感知渲染。
4. 只有在“单层直出 + 效果感知”仍然失败时，才允许升级到更复杂的导出策略。

这条规则的目的，是避免把本来可以直接导出的标准层，误升级成高成本复杂策略。
